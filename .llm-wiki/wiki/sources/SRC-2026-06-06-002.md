# HMAC request signing pattern

## Summary

HMAC request signing is a stateless authentication mechanism that eliminates the need for API key rotation. A shared secret key is established once per consumer at creation time. Instead of sending the secret with every request (as with a traditional API key), the consumer signs the request payload using HMAC-SHA256 and sends the resulting signature as a header. The server looks up the consumer's stored secret (encrypted at rest with AES-256-GCM), recomputes the signature, and verifies they match.

Key security properties: the secret never travels over the wire (only the signature does), request integrity is guaranteed (the body is part of the signed message), replay attacks are prevented via a timestamp that the server checks against a configurable clock skew window (default 5 minutes), and the system is fully stateless — no session, no token issuance, no rotation lifecycle to manage.

This was implemented as the recommended authentication mechanism for Notifier. It sits alongside the legacy Bearer token auth for backward compatibility. The implementation uses three custom HTTP headers (X-Consumer-ID, X-Timestamp, X-Signature) and the HMAC secret is encrypted at rest using AES-256-GCM with a server-side master key.

## Key Takeaways

- HMAC request signing replaces API key rotation — the secret never changes, and auth is per-request
- Secret never travels over the wire; only the signature does
- Timestamp prevents replay attacks (default 5 min clock skew window)
- AES-256-GCM encryption for secrets at rest
- Backward compatible — Bearer token auth still works alongside HMAC
